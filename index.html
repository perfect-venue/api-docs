Introduction

Welcome to the Perfect Venue API docs. This API is available to Perfect Venue partners. If you're interested in building an integration, please reach out to eng@perfectvenue.com and we'll help you get set up.

This site was generated with Redoc and the OpenAPI/Swagger format.

Quick start

Use this flow when building a new integration:

  1. Contact Perfect Venue to provision OAuth access for your integration.
  2. Perfect Venue will configure your OAuth application and provide client credentials to your partner developer team.
  3. Send a Perfect Venue user with access to the relevant venues through the OAuth authorization code flow.
  4. Exchange the authorization code for an access token and refresh token.
  5. Call GET /venues to find the venues available to the authorizing Perfect Venue user.
  6. Store the Perfect Venue venue_id values in your system.
  7. Create or map contacts with POST /contacts, or create a contact inline when creating an event.
  8. Create events with POST /events.
  9. Use GET /events plus webhooks to keep your system in sync.

All authenticated API requests use this base URL:

https://api.perfectvenue.com/api/v1

Implementation path

1. List venues

After a Perfect Venue user authorizes your integration, call GET /venues first. Venue IDs are required when creating contacts and events. The venue response also includes venue-specific configuration such as event types, spaces, and custom fields.

curl https://api.perfectvenue.com/api/v1/venues \
  -H "Authorization: Bearer ACCESS_TOKEN"

Save the id for each venue your integration should write to.

2. Create a contact

Contacts are scoped to a venue. Perfect Venue does not enforce uniqueness on contact email or phone, so store the returned id in your own system if you need to avoid duplicates.

curl https://api.perfectvenue.com/api/v1/contacts \
  -H "Authorization: Bearer ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "contact": {
      "venue_id": "VENUE_ID",
      "first_name": "Ada",
      "last_name": "Lovelace",
      "email": "ada@example.com",
      "phone": "+14155550100"
    }
  }'

3. Create an event

Create an event with a known contact_id, or pass a top-level contact object to create or reuse a contact inline. Use start_time and end_time for new integrations; start_date and end_date are deprecated.

curl https://api.perfectvenue.com/api/v1/events \
  -H "Authorization: Bearer ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "event": {
      "venue_id": "VENUE_ID",
      "contact_id": "CONTACT_ID",
      "name": "Henry's Wedding",
      "start_time": "2026-08-15T18:00:00Z",
      "end_time": "2026-08-15T23:00:00Z",
      "group_size": 120,
      "source": "Partner Integration",
      "event_type": "Wedding",
      "spaces": ["main-hall"],
      "custom": {
        "dietary_restrictions": "vegan"
      }
    },
    "message": {
      "text": "Interested in a Saturday evening reception."
    }
  }'

4. Sync events

Use GET /events to backfill or reconcile data. Always pass venue_id; use pagination for large result sets.

curl "https://api.perfectvenue.com/api/v1/events?venue_id=VENUE_ID&per_page=100&page=1" \
  -H "Authorization: Bearer ACCESS_TOKEN"

For incremental syncs, filter by updated_at:

curl "https://api.perfectvenue.com/api/v1/events?venue_id=VENUE_ID&date_field=updated_at&date_from=2026-06-01T00:00:00Z&date_to=2026-06-30T23:59:59Z" \
  -H "Authorization: Bearer ACCESS_TOKEN"

Data model notes

  • venue_id identifies a venue available to the authorizing Perfect Venue user and is required for creating contacts and events.
    • contact_id identifies the event's primary contact.
    • event_type must match an event type configured for the venue.
    • spaces should contain space slugs from the venue's spaces array.
    • custom values are keyed by the venue's custom field key.
    • Event timestamps are ISO 8601 strings. Use the venue's timezone when displaying local times to users.
    • Money amounts are integer cents, for example 150000 for $1,500.00.

Error handling

Most validation, permission, and not-found errors use status: "fail" and include one or more messages in data.

{
  "status": "fail",
  "data": [
    { "message": "End time must be after start time" }
  ]
}

Common error statuses:

  • 400 Bad Request: validation failed or required parameters are missing.
  • 401 Unauthorized: the access token is missing, invalid, or expired.
  • 403 Forbidden: the token does not grant access to the requested venue or resource.
  • 404 Not Found: the requested resource does not exist or is not visible to the authorizing Perfect Venue user.
  • 429 Too Many Requests: the access token exceeded the rate limit. Rate-limit responses use a problem-details style JSON body with title, status, and detail.

Authentication

The Perfect Venue API uses Bearer tokens for authentication. You can obtain a token by following the OAuth 2.0 authorization code flow, described below.

Authorizing the app

Direct a Perfect Venue user to the Authorization URL with the following query parameters. The user must log in and grant permission before your integration can receive an authorization code.

  • response_type: Set to "code"
  • client_id: Your application's unique client ID, provided by Perfect Venue
  • redirect_uri: The URL to which the user will be redirected after authorization
  • scope: Space-separated list of permissions your application needs
  • state: (Optional but recommended) A random string to maintain state between the request and callback, helping to prevent CSRF attacks

Example Authorization URL:

  https://api.perfectvenue.com/oauth/authorize?response_type=code&client_id=CLIENT_ID&redirect_uri=REDIRECT_URI&scope=venues:read%20events:write&state=RANDOM_STRING

Obtaining an access token

Once the Perfect Venue user has authorized your application, they will be redirected to the specified redirect_uri with a code parameter in the query string.

To exchange this authorization code for an access token, send a POST request to the Token URL with the following setup:

  1. Authorization Header: Use Basic HTTP authentication, where the client_id and client_secret are combined in the format client_id:client_secret and Base64 encoded. This encoded value is included in the Authorization header as Basic <encoded_credentials>.
  2. Body Parameters (URL-encoded)
    • grant_type: Set to "authorization_code"
    • code: The authorization code received in the redirect
    • redirect_uri: Must match the redirect_uri used the previous step

Example Token Request: 

  POST https://api.perfectvenue.com/oauth/token
  Authorization: Basic BASE64_ENCODED(client_id:client_secret)
  Content-Type: application/x-www-form-urlencoded

  grant_type=authorization_code&code=AUTHORIZATION_CODE&redirect_uri=REDIRECT_URI

Using access tokens

Upon a successful token exchange, you’ll receive a JSON response with an access token. Include this token in the Authorization header as Bearer <token> for all authenticated requests.

Access tokens represent the Perfect Venue user who authorized the application. API responses and writes are limited to the venues and resources that user can access in Perfect Venue.

Example API Request:

  GET /api/v1/venues
  Host: api.perfectvenue.com
  Authorization: Bearer YOUR_ACCESS_TOKEN

Using refresh tokens

Token responses will include a refresh token that can be exchanged for a new access token.

To exchange this authorization code for an access token, send a POST request to the Token URL with the following setup:

  1. Authorization Header: Use Basic HTTP authentication, where the client_id and client_secret are combined in the format client_id:client_secret and Base64 encoded. This encoded value is included in the Authorization header as Basic <encoded_credentials>.
  2. Body Parameters (URL-encoded)
    • grant_type: Set to "refresh_token"
    • refresh_token: The refresh token obtained from a previous token request

Example Refresh Token Request: 

  POST https://api.perfectvenue.com/oauth/token
  Authorization: Basic BASE64_ENCODED(client_id:client_secret)
  Content-Type: application/x-www-form-urlencoded

  grant_type=refresh_token&refresh_token=REFRESH_TOKEN

Rate limiting

The API is rate limited to 100 requests per minute per access token. If you exceed this limit, the API will respond with 429 Too Many Requests.

When a request is rate limited, the response includes headers that help you decide when to retry:

Header Description
Retry-After Seconds to wait before retrying
X-RateLimit-Limit Maximum requests allowed per minute
X-RateLimit-Remaining Requests remaining in the current window
X-RateLimit-Reset Unix timestamp when the current limit window resets

When you receive a 429, wait before retrying. For API support, contact eng@perfectvenue.com.

Webhooks

Perfect Venue can notify your application in real time when data changes for venues available to the authorizing Perfect Venue user. Use webhooks for incremental updates, and use the REST API to fetch the current resource state whenever you need to reconcile.

Quick start

Use this flow when adding webhook support:

  1. Build an HTTPS endpoint that can receive POST requests from Perfect Venue.
  2. Send Perfect Venue your endpoint URL and the event types your integration needs.
  3. Perfect Venue will add the webhook endpoint and event subscriptions to your OAuth application.
  4. Store the endpoint signing secret provided during setup.
  5. Verify the PV-Signature header on every delivery.
  6. Use PV-Webhook-Id as your idempotency key.
  7. Return a 2xx response only after you have safely accepted the delivery.
  8. Fetch the latest resource from the REST API when your local copy needs reconciliation.

Subscribing to events

Webhook subscriptions are configured by Perfect Venue during integration setup. To subscribe, send eng@perfectvenue.com:

Field Description
Endpoint URL The HTTPS URL where Perfect Venue should send webhook deliveries
Event types One or more public event type names from the table below
OAuth application The OAuth application Perfect Venue configured for your integration

Example setup request:

Endpoint URL: https://example.com/perfect-venue/webhooks
Event types: event.created, event.status_changed, event.deleted, contact.created
OAuth application: Partner Integration Name

Perfect Venue configures webhook subscriptions on your OAuth application. Your endpoint can subscribe to multiple event types.

Your endpoint must be reachable over HTTPS and should be prepared to receive duplicate deliveries.

Your endpoint receives events only for venues available to users who have authorized your OAuth application.

Delivery request

Each delivery is an HTTPS POST to your configured endpoint.

Required headers:

Header Description
Content-Type Always application/json
PV-Webhook-Id Stable delivery ID. Use this to deduplicate retries.
PV-Signature Timestamped HMAC signature used to verify the request.

Example delivery:

POST /perfect-venue/webhooks HTTP/1.1
Content-Type: application/json
PV-Webhook-Id: f8c1a0e2-6d2f-4a0d-99a2-1fd6f9f3c1aa
PV-Signature: t=1718557200,v1=5257a869e7ecebeda32affa62cdca3fa...

Payload envelope

The JSON body always follows this envelope:

{

Signature verification

Every delivery includes a PV-Signature header:

PV-Signature: t=1718557200,v1=5257a869e7ecebeda32affa62cdca3fa...

t is the Unix timestamp the request was signed. v1 is the HMAC-SHA256 hex digest of "#{msg_id}.#{t}.#{raw_body}" where msg_id is the value of the PV-Webhook-Id header, computed with your endpoint's signing secret.

Verification steps:

  1. Read the raw request body before parsing JSON.
  2. Parse t and v1 from PV-Signature.
  3. Reject requests where t is too far from the current time. We recommend a 5-minute tolerance.
  4. Build the signed payload as PV-Webhook-Id + "." + t + "." + raw_body.
  5. Compute the HMAC-SHA256 hex digest with your endpoint signing secret.
  6. Compare the computed digest to v1 using a constant-time comparison.

Including PV-Webhook-Id in the signed content means a replay attack with a swapped webhook ID will fail verification.

Example Node.js verifier:

const crypto = require("crypto");

function verifyPerfectVenueWebhook({ rawBody, webhookId, signatureHeader, signingSecret }) {
  const parts = Object.fromEntries(
    signatureHeader.split(",").map((part) => part.split("="))
  );
  const timestamp = parts.t;
  const expected = parts.v1;

  const signedPayload = `${webhookId}.${timestamp}.${rawBody}`;
  const actual = crypto
    .createHmac("sha256", signingSecret)
    .update(signedPayload)
    .digest("hex");

  return crypto.timingSafeEqual(
    Buffer.from(actual, "hex"),
    Buffer.from(expected, "hex")
  );
}

Idempotency

The PV-Webhook-Id header (and the id field in the body) is a stable identifier for an event delivery. It does not change across retries. Use it to deduplicate: if you have already processed an id, acknowledge with 2xx and ignore the repeat.

Delivery is at-least-once. Your handler must be safe to run twice for the same PV-Webhook-Id.

Response handling

Return a 2xx status after your endpoint has accepted the delivery. Any non-2xx response, timeout, or connection error is treated as a failed delivery and retried.

Recommended handler behavior:

  1. Verify the signature.
  2. Check whether PV-Webhook-Id has already been processed.
  3. Persist the delivery or enqueue work in your system.
  4. Return 2xx.
  5. Process the event asynchronously when possible.

Ordering and sync

There is no global ordering guarantee. Deliveries are dispatched concurrently and can arrive out of order. Events about the same resource carry a created_at you can order by, and you can always read the current state from the REST API to reconcile. For high-confidence syncs, combine webhooks with periodic GET /events backfills filtered by updated_at.

Retries and endpoint health

Failed deliveries are retried automatically.

If an endpoint fails repeatedly, it may be disabled until it is healthy again.

1.3.0 — 2026-06-18

Webhooks

New Webhooks section

Outgoing webhooks are now available. Register an HTTPS endpoint on your OAuth application and subscribe to the event types you care about. Perfect Venue will POST a signed JSON payload to your endpoint whenever a subscribed event occurs.

Event types available in this release:

Type Fires when
event.created An event is created
event.status_changed An event's status changes
event.deleted An event is deleted
contact.created A new inbound inquiry arrives

See the Webhooks section for the full delivery contract, payload schemas, signature verification, and retry behavior.

Events

New List Events endpoint

GET /events is now available. Returns a paginated list of events for a venue with support for filtering by status, event type, source, spaces, contact, and date range.

Parameter Type Description
venue_id string Required. The ID of the venue
status string[] Filter by status: lead, qualified, proposal_sent, confirmed, balance_due, completed, lost
event_type string[] Filter by event type name
source string[] Filter by event source
spaces string[] Filter by space slug
contact_id string Filter by contact ID
order string Sort field: created_at (default), updated_at, start_time, name
sort_direction string asc or desc (default)
date_from string ISO 8601 start of date range (requires date_to)
date_to string ISO 8601 end of date range (requires date_from)
date_field string Date field for range filter: created_at (default), updated_at, start_time
page integer Page number (default: 1)
per_page integer Results per page, max 100 (default: 25)

1.2.0 — 2026-04-01

Venues

New fields on Get Venue and List Venues

Field Type Description
event_types object[] Event types configured for the venue, each with a name
spaces object[] Spaces available at the venue, each with slug, name, standing, and seated capacity
custom_fields object[] Custom fields configured for the venue, each with key, name, data_type, and optional options

Events

New fields on Create Event and Update Event

The following optional fields are now supported:

Field Type Description
amount integer Estimated event value in cents (e.g. 150000 for $1,500.00)
event_type string Event type name, must match a type configured for the venue
spaces string[] Space slugs to assign to the event
custom object Custom field values keyed by field key

Inline contact creation on Create Event

You can now create a contact inline when creating an event by passing a top-level contact object instead of a contact_id. If a contact with the given email already exists in your organization, it will be reused.

Initial message on Create Event

You can now include an initial message from the contact to the venue when creating an event by passing a top-level message object with a text field.

New Update Event endpoint

PATCH /events/{event_id} is now available. All fields are optional — only the fields you include will be updated.
